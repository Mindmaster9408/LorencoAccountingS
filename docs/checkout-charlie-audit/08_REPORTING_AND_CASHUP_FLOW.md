# 08 — Reporting and Cash-Up Flow

---

## 1. Reporting Architecture

Reports are generated by `Point of Sale/routes/reports.js`.  
All reports:
- Require authentication + company context
- Run real-time SQL aggregations (no pre-computed summary tables used by default)
- Support date range filtering (`startDate`, `endDate` query params)
- Return data suitable for CSV export

## 2. Available Reports

### Gross Profit Report
`GET /api/reports/sales/gross-profit`  
Permission: `REPORTS.PROFIT`

```sql
SELECT
  s.created_at, s.sale_number,
  u.full_name as cashier,
  SUM(si.quantity × si.unit_price) as subtotal,
  SUM(s.vat_amount) as vat,
  s.total_amount,
  SUM((si.unit_price - p.cost_price) × si.quantity) as gross_profit,
  ROUND(100.0 × SUM((si.unit_price - p.cost_price) × si.quantity) / SUM(si.total_price), 2) as profit_margin
FROM sales s
JOIN sale_items si ON s.id = si.sale_id
JOIN products p ON si.product_id = p.id
JOIN users u ON s.user_id = u.id
WHERE s.company_id = ?  [AND date filters]
GROUP BY s.id
ORDER BY s.created_at DESC
```

Returns each sale with its gross profit and margin.  
Summary: `totalSales`, `totalProfit`, `profitMargin`, `transactionCount`

**Risk:** Requires `cost_price` to be set on products. If `cost_price = 0` or NULL, gross profit = unit_price × quantity (misleading). No validation that cost_price is set before showing this report.

### Gross Profit by Salesperson
`GET /api/reports/sales/gross-profit-by-person`  
Permission: `REPORTS.PROFIT`

```sql
SELECT u.full_name as cashier,
  COUNT(DISTINCT s.id) as sales_count,
  SUM(s.total_amount) as total_sales,
  SUM(s.vat_amount) as total_vat,
  SUM((si.unit_price - p.cost_price) × si.quantity) as gross_profit,
  profit_margin
FROM sales s JOIN sale_items si JOIN products p JOIN users u
WHERE s.company_id = ? [AND date filters]
GROUP BY u.id, u.full_name
ORDER BY gross_profit DESC
```

### Daily Sales Summary
`GET /api/reports/sales/daily-summary`

```sql
SELECT
  DATE(s.created_at) as date,
  COUNT(DISTINCT s.id) as transaction_count,
  SUM(s.subtotal) as total_subtotal,
  SUM(s.vat_amount) as total_vat,
  SUM(s.total_amount) as total_sales
FROM sales s
WHERE s.company_id = ? AND s.status = 'completed' [AND date filters]
GROUP BY DATE(s.created_at)
ORDER BY date DESC
```

**Important:** Only `status = 'completed'` sales are counted. Voided sales are excluded from totals.

## 3. Cash-Up Flow

### UI Flow

1. Cashier navigates to Cash Up tab
2. Open session loaded: `GET /api/pos/sessions?status=open`
3. Session sales loaded: `GET /api/pos/sessions/:id/sales`
4. Expected cash calculated:

```javascript
expectedBalance = openingBalance + totalCashSales
// Note: CARD sales do NOT add to expected cash (physical till only)
```

5. Denomination grid shown:
```
R200 × qty = R___
R100 × qty = R___
R50  × qty = R___
R20  × qty = R___
R10  × qty = R___
R5   × qty = R___
R2   × qty = R___
R1   × qty = R___
50c  × qty = R___
20c  × qty = R___
10c  × qty = R___
5c   × qty = R___
────────────────
Total       R___
```

6. App auto-totals as cashier enters denominations
7. Variance displayed: `actual_cash - expected_cash`
8. Notes field for cashier to document discrepancies
9. Submit: `POST /api/pos/sessions/:id/close`

### Session Close API

`POST /api/pos/sessions/:id/close`  
Body: `{ closingBalance, closing_balance, expectedBalance, expected_balance, variance, notes }`  
(supports both camelCase and snake_case variants)

```javascript
// Server-side calculation:
const calculatedExpected = parseFloat(session.opening_balance) + parseFloat(session.total_sales);
const finalExpected = expectedBal || calculatedExpected;
const finalVariance = varianceVal !== undefined ? varianceVal : (parseFloat(closingBal) - finalExpected);

UPDATE till_sessions
SET closing_balance = ?, expected_balance = ?, variance = ?,
    status = 'closed', closed_at = CURRENT_TIMESTAMP, notes = ?
WHERE id = ? AND company_id = ?
```

**Important:** `session.total_sales` is calculated via:
```sql
SELECT COALESCE(SUM(s.total_amount), 0) as total_sales
FROM till_sessions ts
LEFT JOIN sales s ON s.till_session_id = ts.id
WHERE ts.id = ? AND ts.company_id = ?
```

This includes ALL sales (cash AND card AND EFT etc.) in the expected balance calculation. If a store takes card payments, the expected cash does not deduct card sales. This means the variance calculation is only accurate if the cashier manually accounts for card terminal totals.

## 4. Variance Recording

Variance is stored in `till_sessions.variance`:
- Positive variance = over (more cash than expected) — could indicate error in change calculation or unrecorded adjustment
- Negative variance = short (less cash than expected) — could indicate theft, giving too much change, or missed cash sale

The `cash_variances` and `employee_variance_summary` tables exist for loss prevention tracking:
```sql
cash_variances (company_id, location_id, till_session_id, user_id, variance_date,
  expected_amount, actual_amount, variance_amount, variance_reason, is_investigated)

employee_variance_summary (company_id, user_id, period_start, period_end,
  variance_count, total_positive_variance, total_negative_variance, void_count, refund_count)
```

These are populated by the loss prevention module routes — not automatically by the cash-up flow itself.

## 5. VAT / Tax Implications in Reports

**VAT Rate:**  
Stored in `company_settings.vat_rate` (default 15%) and `vat_settings.vat_rate`.  
Product-level VAT: `products.requires_vat` and `products.vat_rate`.

**Legacy VAT calculation (in sale):**
```
VAT = subtotal × 0.15   ← External (exclusive) VAT
```
This is mathematically equivalent to saying all prices are VAT-exclusive. The `vat_amount` stored on the sale is this computed value.

**Ecosystem VAT calculation (in sale):**
```
VAT = linePrice × (vat_rate / (100 + vat_rate))   ← Inclusive VAT extracted
```
This assumes prices INCLUDE VAT already. The `vat_amount` is the extracted portion.

**VAT reports pull from `sales.vat_amount`** — so the report accuracy depends entirely on which calculation method was used when the sale was created.

**For SARS VAT201 reporting:** The VAT summary report provides daily totals of VAT collected. Whether the VAT201 figures are correct depends on the VAT calculation method. If inconsistent between backends, the figures will be unreliable for tax submission.

## 6. Analytics Dashboard (Enterprise)

`routes/analytics.js` provides additional analytics endpoints:

```
GET /api/analytics/dashboard     ← KPI summary cards
GET /api/analytics/kpis          ← Target vs actual KPI comparison
GET /api/analytics/trends        ← Sales trend data for charts
GET /api/analytics/export        ← Data export
```

These use the `daily_sales_summary` and `hourly_sales_summary` pre-aggregated tables when available. Pre-aggregation is NOT automatic — it must be triggered or run by a scheduled job.

## 8. CSV Export

All reports support CSV export. The export is handled client-side in the frontend:
- API returns JSON data
- Frontend converts to CSV string
- `<a download>` link triggered programmatically

No server-side PDF or Excel generation exists currently.
