# Codebox 47 — Practice SOP Templates + Workflow Instruction Library

> Module: Practice Management — SOP Library
> Status: Complete (migration 105 not yet applied to Supabase)
> Migration: 105
> Routes: `/api/practice/sop/*`

---

## Purpose

The practice's operational instruction manual — reusable SOP templates describing
**HOW** work must be performed. Attaches to workflow templates, workflow steps,
tasks, review tasks, compliance packs, completion packs, and knowledge articles
so staff can open the relevant procedure directly from the work they're doing.

**This is NOT AI. NOT document management. NOT workflow execution.**
The Knowledge Base (Codebox 46) stores *what to know*; this module stores
*how to do it*.

## Migration 105

### `practice_sop_templates` — one row per SOP

| Field | Notes |
|---|---|
| `category` | Free TEXT — **not** enum-constrained (unlike Knowledge Base's `category`). Spec listed no "Allowed" values for this field. |
| `status` | 4-value CHECK: `draft`, `under_review`, `approved`, `archived` (patched post-launch — see "Status Machine" below) |
| `difficulty` | Nullable, CHECK `IS NULL OR IN ('beginner','intermediate','advanced')` |
| `requires_review` | BOOLEAN, default `TRUE` |
| `version` | Integer, starts at 1; bumped on content edits to an approved SOP |
| `estimated_minutes` | Nullable integer |

Auto-updates `updated_at` via `tg_pst_updated_at` trigger. `id` uses `GENERATED BY DEFAULT AS IDENTITY` (matches Codebox 46 convention).

### `practice_sop_links` — SOP ↔ record attachments

`linked_type` (7-value CHECK): `workflow_template`, `workflow_step`, `task`, `review_task`, `compliance_pack`, `completion_pack`, `knowledge_article`.

`sort_order` supports ordering multiple SOPs attached to the same record.

`UNIQUE (sop_id, linked_type, linked_id)` prevents duplicate attachments at the DB level (not explicitly requested in the spec, added for consistency with Codebox 46's link table and backed by an app-level 409).

### `practice_sop_events` — append-only audit log

Never updated or deleted.

## Backend — `practice-sop.js`

### Endpoints (13 total — 12 from spec + 1 addition, see below)

| Method | Route | Purpose |
|---|---|---|
| GET | `/summary` | Status counts (draft/under_review/approved/archived) |
| GET | `/linked/:linkedType/:linkedId` | **Addition** — SOPs attached to a given record (see note below) |
| GET | `/` | Paginated list with filters: search, category, status, difficulty |
| POST | `/` | Create SOP (always starts as `draft`) |
| GET | `/:id` | Single SOP |
| PUT | `/:id` | Update — see versioning rule |
| DELETE | `/:id` | Soft archive |
| PUT | `/:id/submit-review` | Status: draft → under_review |
| PUT | `/:id/approve` | Status: under_review → approved |
| GET | `/:id/links` | List attached records |
| POST | `/:id/links` | Attach to a record (ownership-checked where the table is known) |
| DELETE | `/:id/links/:linkId` | Remove an attachment |
| GET | `/:id/events` | Append-only event log |

**Addition beyond the literal spec:** `GET /linked/:linkedType/:linkedId` was not in the spec's endpoint table, but the "Workflow Integration" section requires showing "Attached SOPs" / an "Instruction button" / "Standard Procedure" / "Procedure" on five different record types — none of which is possible without a reverse lookup from record → SOPs. This mirrors the identical endpoint added for the Knowledge Base in Codebox 46, kept the same shape for consistency.

### Status Machine — 4 values (patched)

```
draft → under_review → approved → archived
  (any non-archived) → archived (also allowed directly)
```

Only `archived` is terminal. `PUT /:id` and `POST /:id/links` are blocked once archived. This now matches the Knowledge Base's 4-state machine exactly.

### Submit-Review / Approve Semantics (patched — supersedes original launch design)

**As originally shipped**, the spec's field list only listed 3 status values (`draft`, `approved`, `archived`) with no `under_review` state, so `submit-review` logged an audit event without changing status, and `approve` transitioned `draft → approved` directly. This ambiguity was flagged as a follow-up in the original handoff.

**Patched design (current):** `under_review` was added to the CHECK constraint. `PUT /:id/submit-review` now validates the SOP is `draft` and transitions it to `under_review`. `PUT /:id/approve` now validates the SOP is `under_review` and transitions it to `approved` (setting `reviewed_at`/`reviewed_by`/`approved_at`/`approved_by` together, same as before). The frontend footer was updated so `draft` shows only "Submit for Review" and `under_review` shows only "Approve" (previously both buttons showed simultaneously on `draft`).

Since migration 105 had not yet been applied to Supabase at the time of this patch, the CHECK constraint was edited directly in the same migration file rather than adding a new migration — there was no live table state to alter.

### Versioning Rule (identical pattern to Knowledge Base)

Editing an **approved** SOP's content-affecting fields (`title`, `summary`, `instruction_body`, `category`) via `PUT /:id`:
- Bumps `version` by 1
- Resets `status` to `draft`
- Clears `approved_at` / `approved_by`
- Requires `submit-review` → `approve` again

No separate version-history table — only the current row and its version number are stored (see follow-up notes).

### Linked-Record Ownership Check

`_verifyLinkedRecordOwnership` maps all 7 `linked_type` values to real tables: `practice_workflow_templates`, `practice_workflow_template_steps`, `practice_tasks` (used for both `task` and `review_task` — there is no separate review-task table; review tasks are `practice_tasks` rows with `review_required = true`), `practice_compliance_packs`, `practice_tax_completion_packs` (note: SOP's enum value is `completion_pack`, not `tax_completion_pack` like Knowledge Base's), `practice_knowledge_articles`.

### Search Logic

`ilike` on `title`, `summary`, `instruction_body`. No AI search, per spec.

### Multi-Tenant Safety

Every query scoped to `req.companyId`. `_verifySop` and `_verifyLink` re-check ownership before every mutating action.

## Workflow Integration

All five integration points use the same small, additive "jump to SOP page" link pattern established in Codebox 46 — no existing UI was rewritten, only new links/buttons appended:

| Location | File(s) | What was added |
|---|---|---|
| Workflow Template | `workflow-template.html`, `js/workflow-editor.js` | "Attached SOPs ↗" link in the page header, shown only once the template has an ID (edit mode); populated in `loadTemplate()` |
| Workflow Step | `js/workflow-editor.js` (`renderStepsList`) | "📋 Instructions" link per step card, linking to `?linked_type=workflow_step&linked_id=<step.id>` |
| Task Detail | `js/tasks.js` (task card header) | "📋 Procedure" link per task card, linking to `?linked_type=task&linked_id=<task.id>` |
| Compliance Pack | `js/compliance-packs.js` (`renderDetailHeader`) | "Procedure ↗" link injected via DOM (no HTML file change needed) after the `cpDSub` element |
| Completion Pack | `js/tax-completion.js` (`_renderFooter`) | "Procedure ↗" link appended to the existing footer, alongside the Codebox 46 "Knowledge ↗" link |

All links navigate to `practice-sop.html?linked_type=X&linked_id=Y`, which renders an info banner listing attached SOPs for that record (via `GET /linked/:linkedType/:linkedId`), each clickable to open its detail modal. Attaching an SOP the other direction (from the SOP side) is done via the SOP's Links tab → "+ Attach to Record" → select `linked_type` + record ID.

### `index.js` Mount

```javascript
const practiceSopRouter = require('./practice-sop');
router.use('/sop', practiceSopRouter);
```

Mounted after `/knowledge`, before the `/dashboard` sub-router block.

### `layout.js` Nav

"Practice SOPs" entry added between "Knowledge Base" and "Tax Config".

## Frontend — `practice-sop.html` + `js/practice-sop.js`

### Page

- Summary cards (Draft, Under Review, Approved, Archived) — clickable to filter
- Filter bar: category (free text), status (incl. Under Review), difficulty, free-text search
- Paginated SOP table (status, title, category, difficulty, estimated minutes, version, updated)
- Create SOP modal (title, category, difficulty, estimated minutes, requires review, summary, instructions, effective from/to, internal notes) — always creates as `draft`
- SOP Detail modal — 3 tabs: **Instruction** (full metadata + summary + instructions + internal notes), **Links** (attached records + add/remove), **History** (append-only event log)
- Edit modal — warns before saving if the SOP is `approved` (edit bumps version and returns to draft)
- Attach SOP modal — select `linked_type` (7 values) + record ID + sort order + notes
- Context-sensitive footer: `draft` → Submit for Review; `under_review` → Approve; both plus `approved` → Edit + Archive

### No localStorage / KV

Zero browser storage usage in any new file. `node --check` passed on all new/modified JS files.
