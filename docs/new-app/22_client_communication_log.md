# Codebox 22 — Client Communication Log + Follow-Up Notes Foundation

**App:** Lorenco Practice Management  
**Codebox:** 22 of ±80  
**Status:** Complete  
**Date:** 2026-06-20

---

## Purpose

Build a proper communication history per client. The practice now has health risks, actions, reminders, deadlines, tasks, and engagements — but no way to log what was actually said or agreed with the client. This codebox adds a manual communication log (calls, email notes, WhatsApp notes, meetings, document requests, SARS/CIPC follow-ups) with response tracking and follow-up creation.

**This is manual logging only.** No email sending. No WhatsApp integration. No Gmail/Outlook integration. No Sean AI. No cron.

## What Was Built

### Database — migration 072

```sql
CREATE TABLE practice_client_communications (
    id                       INTEGER      PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
    company_id               INTEGER      NOT NULL,
    client_id                INTEGER      NOT NULL,
    communication_type       TEXT         NOT NULL,
    direction                TEXT         NOT NULL DEFAULT 'outbound',
    subject                  TEXT         NOT NULL,
    body                     TEXT         NULL,
    contact_person_id        INTEGER      NULL,   -- reference-only, no FK
    contact_name             TEXT         NULL,
    contact_email            TEXT         NULL,
    contact_phone            TEXT         NULL,
    related_task_id          INTEGER      NULL,   -- reference-only, no FK
    related_deadline_id      INTEGER      NULL,   -- reference-only, no FK
    related_engagement_id    INTEGER      NULL,   -- reference-only, no FK
    related_reminder_id      INTEGER      NULL,   -- reference-only, no FK
    related_health_action_id INTEGER      NULL,   -- reference-only, no FK
    response_required        BOOLEAN      NOT NULL DEFAULT false,
    response_due_date        DATE         NULL,
    response_status          TEXT         NOT NULL DEFAULT 'not_required',
    responded_at             TIMESTAMPTZ  NULL,
    responded_by             INTEGER      NULL,
    assigned_team_member_id  INTEGER      NULL,
    communication_date       TIMESTAMPTZ  NOT NULL DEFAULT now(),
    is_internal              BOOLEAN      NOT NULL DEFAULT false,
    visibility               TEXT         NOT NULL DEFAULT 'practice',
    attachments_note         TEXT         NULL,
    tags                     JSONB        NOT NULL DEFAULT '[]',
    settings                 JSONB        NOT NULL DEFAULT '{}',
    cancelled_at             TIMESTAMPTZ  NULL,   -- soft delete
    created_at               TIMESTAMPTZ  NOT NULL DEFAULT now(),
    updated_at               TIMESTAMPTZ  NOT NULL DEFAULT now(),
    created_by               INTEGER      NULL,
    updated_by               INTEGER      NULL
);
```

13 indexes including partial indexes on `cancelled_at IS NULL` for active records, `response_status = 'waiting'` for open responses, and compound `(company_id, client_id, communication_date DESC)` for the per-client history query.

**No FK constraints on any `related_*` or `contact_person_id` fields** — reference-only to prevent orphan failures.

### Allowed Values

**communication_type:** `call` | `email_note` | `whatsapp_note` | `meeting` | `document_request` | `sars_followup` | `cipc_followup` | `billing_followup` | `general_note` | `internal_note`

**direction:** `inbound` | `outbound` | `internal`

**response_status:** `not_required` | `waiting` | `received` | `overdue` | `cancelled`

**visibility:** `practice` | `manager_only` | `partner_only`

**Soft delete:** `cancelled_at TIMESTAMPTZ NULL` — hard deletes are never performed. `GET /` excludes rows where `cancelled_at IS NOT NULL`.

### Overdue Detection

`response_status = 'overdue'` is a computed/virtual state. The DB stores `'waiting'`. The backend adds `effective_response_status: 'overdue'` when `response_status = 'waiting' AND response_due_date IS NOT NULL AND response_due_date < today`. The frontend uses `effective_response_status` for display.

### Backend Endpoints (`backend/modules/practice/communications.js`)

Mounted at `/api/practice/communications`.

| Method | Path | Purpose |
|--------|------|---------|
| GET | `/summary` | Aggregate: total, waiting_responses, overdue_responses, document_requests_open, this_week |
| GET | `/` | List with filters (type, direction, response_status, client, assignee, due range, search). Enriched with client_name + assignee_name. |
| POST | `/` | Create communication. Auto-sets response_status='waiting' when response_required=true. Verifies all linked records belong to company. |
| GET | `/:id` | Single communication (including cancelled) |
| POST | `/:id/create-reminder` | Creates `practice_reminders` record; links back via `related_reminder_id` if not already set |
| POST | `/:id/create-task` | Creates `practice_tasks` record; links back via `related_task_id` if not already set |
| PUT | `/:id/mark-responded` | Sets response_status='received', responded_at, responded_by |
| PUT | `/:id/cancel-response` | Sets response_status='cancelled', response_required=false |
| PUT | `/:id` | Generic field update (type, direction, subject, body, contacts, response fields, links, visibility, etc.) |
| DELETE | `/:id` | Soft cancel (sets cancelled_at = now) |

**Route ordering:** `GET /summary` (literal) defined before `GET /:id` (wildcard) to prevent Express collision. All 3-segment PUT routes (`/:id/mark-responded`, `/:id/cancel-response`) defined before 2-segment `PUT /:id`.

### Multi-Tenant Safety

- `req.companyId` from JWT used on all queries — never accepted from request body
- `verifyBelongsToCompany(cid, table, id)` — generic helper called in parallel for all 7 optional linked fields on POST
- `verifyCommOwnership(cid, commId)` — verifies communication belongs to company and is not cancelled before any update
- Text search is server-side filtered after Supabase fetch (avoids n+1 and keeps multi-tenant clean)

### Create-Reminder Flow

`POST /:id/create-reminder`:
1. Verifies communication belongs to company
2. Inserts into `practice_reminders` with `reminder_type='general'`, `source_type='system'`, inherits `client_id` and `assigned_team_member_id` from communication
3. If `comm.related_reminder_id` is null, updates it with new reminder ID
4. Audit log: `communication_reminder_created`

### Create-Task Flow

`POST /:id/create-task`:
1. Verifies communication belongs to company
2. Inserts into `practice_tasks` with `type='general'`, `status='open'`, all required review/approval/qa fields set to safe defaults
3. If `comm.related_task_id` is null, updates it with new task ID
4. Audit log: `communication_followup_task_created`

## Architecture Notes

**No FK constraints on linked fields:** `related_task_id`, `related_deadline_id`, `related_engagement_id`, `related_reminder_id`, `related_health_action_id`, `contact_person_id` are all reference-only. If the linked record is deleted, the communication row survives with a stale reference. This is intentional — communications are historical records.

**Soft delete pattern:** `DELETE /:id` sets `cancelled_at = now()`. `GET /` excludes `WHERE cancelled_at IS NOT NULL`. `GET /:id` returns the record regardless of cancelled state (for history audit). Rows are never physically deleted.

**Overdue is virtual:** `response_status` in DB is always `'waiting'` for open responses. The backend computes `effective_response_status = 'overdue'` when `response_due_date < today`. No cron needed to flip the status.

**client-detail.js integration pattern:** The communication functions use `await PracticeAPI.fetch(...); await res.json()` — matching the older Response-object pattern used in client-detail.js. The newer IIFE pages (reminders.js, communications.js) use the auto-parsed pattern. Both work with the same underlying API.
