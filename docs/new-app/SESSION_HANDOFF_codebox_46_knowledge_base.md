## What Was Built

### Migration 104

Creates three tables — all `IF NOT EXISTS`, safe to re-run:

- **`practice_knowledge_articles`**: One row per knowledge article. 4-status machine (`draft → under_review → approved → archived`). `category` (14-value CHECK), `article_type` (9-value CHECK), `tags` JSONB array, `version` integer (starts 1), `settings` JSONB. Auto-`updated_at` trigger. 5 indexes (company_id, category, article_type, status, title). Uses `GENERATED BY DEFAULT AS IDENTITY` for `id` per the spec (not `SERIAL`).
- **`practice_knowledge_links`**: Links an article to one of 13 record types (`client`, `taxpayer_profile`, `individual_tax_return`, `company_tax_return`, `provisional_tax_plan`, `tax_submission`, `sars_statement_line`, `tax_dispute`, `tax_completion_pack`, `workflow_run`, `task`, `document_request`, `compliance_pack`). `UNIQUE (article_id, linked_type, linked_id)` prevents duplicate links.
- **`practice_knowledge_events`**: Append-only audit log. Never updated or deleted.

### Backend — `knowledge-base.js` (14 endpoints)

Key behaviours:

**Approved-article edit → new version (low-risk versioning, no separate version-history table):**
- `PUT /:id` on an `approved` article: if `title`/`summary`/`content`/`category`/`article_type` change, `version` is bumped, `status` resets to `draft`, `approved_at`/`approved_by` cleared
- Editing `draft`/`under_review` articles does not touch version or status

**Linked-record ownership check:**
- `_verifyLinkedRecordOwnership` maps all 13 `linked_type` values to their real practice tables (confirmed against migrations 057–103) and confirms the record belongs to the requesting company before `POST /:id/links` succeeds
- Returns 404 if the table is known and the record isn't found for this company

**Search:**
- `ilike` across `title`, `summary`, `content`, `source_reference`
- Tag filter: simple case-insensitive exact match against the JSONB tags array, applied in-memory (explicitly NOT AI search per spec)

**Multi-tenant:** Every query scoped to `req.companyId`. `_verifyArticle` + `_verifyLink` re-verify ownership before every mutating action.

**Audit:** `_writeEvent` writes to `practice_knowledge_events` (append-only) on every lifecycle change. `auditFromReq` writes to the shared ecosystem audit trail. Event types logged: `knowledge_article_created`, `knowledge_article_updated`, `knowledge_article_submitted_review`, `knowledge_article_approved`, `knowledge_article_archived`, `knowledge_article_linked`, `knowledge_article_unlinked`.

### Frontend — `knowledge-base.html` + `js/knowledge-base.js` (kb prefix)

- Summary cards (4): Draft, Under Review, Approved, Archived — clickable to filter
- Filter bar: category, type, status, tag, free-text search
- Article table: type badge, status pill, title, category, tag chips, version, updated timestamp
- Create Article modal — always creates as draft; full field set (title, category, type, summary, content, tags, source reference, effective from/to, internal notes)
- Detail modal — 3 tabs:
  - **Content**: metadata grid (type, status, category, version, tags, source ref, effective dates, reviewed/approved timestamps) + summary + full content + internal notes
  - **Links**: linked records list with remove button + "Link to Record" button opening the link modal
  - **Events**: full append-only audit log
- Edit modal: pre-fills all fields; shows an inline warning if editing an approved article (version bump + return to draft)
- Link Article modal: select from all 13 `linked_type` values + record ID + notes
- Context-sensitive footer: Edit (always, unless archived) → Submit for Review (draft) → Approve (under_review) → Archive (always, unless archived)
- `?linked_type=X&linked_id=Y` URL params render an info banner listing already-linked articles for that record (fetched via `GET /linked/:type/:id`), each clickable to open detail
