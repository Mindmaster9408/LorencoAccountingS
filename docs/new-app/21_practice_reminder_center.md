**App:** Lorenco Practice Management  
**Codebox:** 21 of ±80

## Purpose

Central in-app reminder hub for the practice manager. Surfaces actionable items across all clients — overdue deadlines, waiting reviews, stale billing packs, queued periods, health actions, and more — in one prioritised view. No email, no SMS, no push, no cron. In-app only.

## What Was Built

### Database — migration 071

```sql
CREATE TABLE practice_reminders (
    id                       INTEGER      PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
    company_id               INTEGER      NOT NULL,
    reminder_type            TEXT         NOT NULL,  -- see REMINDER_TYPES below
    source_type              TEXT         NOT NULL DEFAULT 'system',
    source_id                INTEGER      NULL,      -- reference-only, no FK
    client_id                INTEGER      NULL,
    assigned_team_member_id  INTEGER      NULL,
    title                    TEXT         NOT NULL,
    message                  TEXT         NULL,
    severity                 TEXT         NOT NULL DEFAULT 'normal',
    action_url               TEXT         NULL,
    metadata                 JSONB        NOT NULL DEFAULT '{}',
    status                   TEXT         NOT NULL DEFAULT 'open',
    due_date                 DATE         NULL,
    snoozed_until            TIMESTAMPTZ  NULL,
    dismissed_at             TIMESTAMPTZ  NULL,
    dismissed_by             INTEGER      NULL,
    completed_at             TIMESTAMPTZ  NULL,
    completed_by             INTEGER      NULL,
    created_at               TIMESTAMPTZ  NOT NULL DEFAULT now(),
    updated_at               TIMESTAMPTZ  NOT NULL DEFAULT now(),
    created_by               INTEGER      NULL
);
```

10 indexes: company, status (partial non-cancelled), type, source dedup (partial open/snoozed), client, assignee, due_date, snoozed_until, severity, source_type.

**No FK constraints on source_id** — reference-only to prevent orphan failures if the source record is deleted.

### Allowed Values

**reminder_type:** `deadline_due` | `deadline_overdue` | `review_waiting` | `approval_waiting` | `billing_waiting` | `health_action` | `period_waiting` | `engagement_setup` | `capacity_warning` | `general`

**source_type:** `deadline` | `task` | `billing_pack` | `health_action` | `engagement_period` | `engagement` | `team_member` | `client` | `system`

**severity:** `low` | `normal` | `high` | `urgent`

**status:** `open` | `snoozed` | `completed` | `dismissed` | `cancelled`

### Backend Endpoints (`backend/modules/practice/reminders.js`)

Mounted at `/api/practice/reminders`.

| Method | Path | Purpose |
|--------|------|---------|
| GET | `/summary` | Aggregate counts: open, overdue, due_today, due_this_week, urgent, snoozed |
| GET | `/suggestions` | Scan live DB and return actionable items with no open reminder |
| GET | `/` | List reminders with filters (status, type, severity, client, assignee, due range) |
| POST | `/create-from-suggestion` | Create reminder from a suggestion (dedup check + source ownership check) |
| POST | `/` | Create manual reminder |
| PUT | `/:id/snooze` | Snooze: set status='snoozed', set snoozed_until (must be future datetime) |
| PUT | `/:id/complete` | Mark completed |
| PUT | `/:id/dismiss` | Mark dismissed |
| PUT | `/:id` | Generic field update (title, message, severity, due_date, etc.) |
| DELETE | `/:id` | Soft cancel (sets status='cancelled', row is preserved) |

**Route ordering safety:** `/summary`, `/suggestions`, `/create-from-suggestion` (all literal) are defined BEFORE `/:id` (wildcard) to prevent Express wildcard collision.

### Suggestions Engine (`GET /suggestions`)

Runs 10 parallel Supabase queries and returns items from live operational data that have no existing open/snoozed reminder.

| Source | Query | reminder_type | severity |
|--------|-------|---------------|----------|
| `practice_deadlines` | `due_date < today`, not completed/cancelled | `deadline_overdue` | high (urgent if >7 days) |
| `practice_deadlines` | `today <= due_date <= today+7`, not completed/cancelled | `deadline_due` | high if ≤2 days, else normal |
| `practice_tasks` | `review_status IN (pending, in_review)`, not completed/cancelled | `review_waiting` | normal |
| `practice_tasks` | `approval_status IN (pending, in_review)`, not completed/cancelled | `approval_waiting` | normal |
| `practice_billing_packs` | `status IN (draft, reviewed)` | `billing_waiting` | high if >30 days old |
| `practice_client_health_actions` | `action_status IN (open, in_progress)`, `due_date <= today` | `health_action` | high if overdue |
| `practice_engagement_periods` | `status IN (queued, ready)`, `period_start < today` | `period_waiting` | normal |
| `practice_client_engagements` | `status=active`, `responsible_team_member_id IS NULL` | `engagement_setup` | normal |
| `practice_client_engagements` | `status=active`, recurring type, missing recurrence config | `engagement_setup` | low |
| `practice_team_members` + `practice_tasks` | team member where taskHours > weekly_capacity_hours | `capacity_warning` | high if >120% |

**Duplicate prevention:** A local `Set` of `"source_type:source_id:reminder_type"` keys from existing open/snoozed reminders prevents showing a suggestion that already has an active reminder. A second local `addedKeys` Set prevents the same key appearing twice in one suggestions response (e.g. an engagement with both "no owner" and "missing recurrence" only appears once).

### Multi-Tenant Safety

- All DB queries use `req.companyId` from JWT — never from request body.
- `verifyReminderOwnership(cid, reminderId)` — verifies reminder belongs to this company before any update.
- `verifySourceOwnership(cid, source_type, source_id)` — verifies source record belongs to this company before create-from-suggestion.
- `duplicateExists(cid, source_type, source_id, reminder_type)` — prevents duplicate open reminders per source item.

### Frontend (`reminders.html` + `js/reminders.js`)

**IIFE module pattern.** No business data in `localStorage` (Rule D compliant).

**Page structure:**
- 6 summary cards (open, overdue, due today, due this week, urgent, snoozed)
- Filter toolbar (status, type, severity, text search)
- Reminder list table with severity dot, type badge, status badge
- Suggestions panel (expandable — loads on demand, not on page load)
- Create modal (type, severity, client, assignee, due date, message)
- Snooze modal (quick options: tomorrow, 3 days, next week, 2 weeks + manual datetime)

**State:**
- `_allReminders` — loaded once, filtered client-side
- `_suggestions` — loaded on demand when suggestions panel is opened
- `_clients`, `_members` — loaded once for modal pickers
- `_snoozingId`, `_submitting` — action guards

**Suggestion flow:**
1. User clicks "💡 Suggestions" → panel opens
2. `GET /suggestions` fired (10 parallel queries on backend)
3. Items rendered with `→ Go` link and `+ Remind` button
4. Clicking `+ Remind` calls `POST /create-from-suggestion`
5. Created reminder appears in main list, suggestion removed from panel

**Snooze flow:**
1. User clicks 💤 on a reminder
2. Snooze modal opens — quick options or manual datetime
3. Submit calls `PUT /:id/snooze` with `snoozed_until` as ISO timestamp
4. Reminder status changes to `snoozed`, shows "Snoozed until X" in row

## Architecture Notes

**Why suggestions load on demand:** The `/suggestions` endpoint fires 10 parallel Supabase queries (deadlines, tasks ×3, billing packs, health actions, periods, engagements, team members, capacity tasks). Loading this on every page load would be wasteful. The panel is collapsed by default and loads only when the user opens it.

**Why no FK on source_id:** Health actions, deadlines, tasks, and other source records can be deleted independently. A FK would cause the reminder row to become orphaned or violate constraints. The reference is informational only — if the source record is gone, the reminder survives.

**Soft delete pattern:** `DELETE /:id` sets `status='cancelled'` — rows are never physically deleted. This preserves audit history.

**Capacity warning computation:** Uses `capacity_is_active` (set in migration 068) rather than `is_active` for the team member filter. Only members with `weekly_capacity_hours > 0` and `capacity_is_active = true` are included in the overload check.

## Follow-Up Notes

| Item | Impact |
|------|--------|
| `action_url` is a static path — when a deadline or task is deleted, the link goes to a list page, not a specific record | Low — list pages work fine; deep-linking could be added later |
| Suggestions cap at 200/100 per category (backend `.limit()`) — very large practices may not see all items | Low — adequate for current scale |
| No "re-open from snoozed" button — user can complete or dismiss a snoozed reminder, but not unsnooze back to open without snooze modal | Low — acceptable UX; add a "Re-open" button in a future codebox if needed |
| Capacity warning uses estimated_hours sum vs weekly_capacity_hours — does not account for period boundaries | Low — same approximation used in capacity.html |
