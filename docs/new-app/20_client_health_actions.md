# Codebox 20 — Client Health Alerts + Operational Follow-up Actions

**App:** Lorenco Practice Management  
**Codebox:** 20 of ±80  
**Status:** Complete  
**Date:** 2026-06-20  

---

## Purpose

Turn client health from visibility into action. After Codebox 19 gave the practice a risk score per client, Codebox 20 lets users click on a risk factor and create a trackable follow-up action — with optional task creation, assignee, due date, and status tracking. No AI, no automation, no cron.

---

## What Was Built

### Database (migration 070)

```sql
CREATE TABLE practice_client_health_actions (
    id                       INTEGER  PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
    company_id               INTEGER  NOT NULL,
    client_id                INTEGER  NOT NULL,
    health_snapshot_id       INTEGER  NULL,
    action_type              TEXT     NOT NULL,
    action_title             TEXT     NOT NULL,
    action_status            TEXT     NOT NULL DEFAULT 'open',
    source_risk_code         TEXT     NULL,
    source_risk_label        TEXT     NULL,
    linked_task_id           INTEGER  NULL,
    linked_deadline_id       INTEGER  NULL,
    linked_period_id         INTEGER  NULL,
    linked_billing_pack_id   INTEGER  NULL,
    assigned_team_member_id  INTEGER  NULL,
    due_date                 DATE     NULL,
    completed_at             TIMESTAMPTZ NULL,
    completed_by             INTEGER  NULL,
    notes                    TEXT     NULL,
    settings                 JSONB    NOT NULL DEFAULT '{}',
    created_at               TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at               TIMESTAMPTZ NOT NULL DEFAULT now(),
    created_by               INTEGER  NULL,
    updated_by               INTEGER  NULL
);
```

No FK constraints on linked_* fields — reference-only to prevent orphan failures.

### Allowed Values

**action_type:** `create_task` | `assign_owner` | `open_deadline` | `generate_period` | `review_wip` | `fix_recurrence` | `fix_missing_owner` | `general_followup`

**action_status:** `open` | `in_progress` | `completed` | `dismissed` | `cancelled`

### Backend Endpoints (`backend/modules/practice/client-health.js`)

| Method | Path | Purpose |
|--------|------|---------|
| GET | `/api/practice/client-health/actions/summary` | Company-wide action counts |
| GET | `/api/practice/client-health/:clientId/actions` | List client's actions |
| POST | `/api/practice/client-health/:clientId/actions/from-risk` | Create from a risk factor (optionally creates linked task) |
| POST | `/api/practice/client-health/:clientId/actions` | Create manual action |
| PUT | `/api/practice/client-health/actions/:id/complete` | Mark completed |
| PUT | `/api/practice/client-health/actions/:id/dismiss` | Mark dismissed |
| PUT | `/api/practice/client-health/actions/:id` | Generic field update |

### Task Creation Logic (`from-risk` with `create_task`)

When `preferred_action_type = create_task`:
1. Inserts row into `practice_tasks` with `type='general'`, `status='open'`, `review_required=false`
2. `description` is auto-set: `"Created from Client Health. Risk: <risk_label>"`
3. `preparer_team_member_id` is set to the assignee if provided
4. Returns `linked_task_id` in the response
5. Inserts action row with `linked_task_id` set
6. Two audit log entries: `health_action_task_created` + `health_action_created`

### Multi-Tenant Safety

- `verifyClientBelongsToCompany(cid, clientId)` called on every write that takes a `clientId` param
- `verifyActionBelongsToCompany(cid, actionId)` called on every action update/complete/dismiss
- `verifyTeamMemberLocal(cid, memberId)` called when `assigned_team_member_id` is provided
- `req.companyId` from JWT is used for all DB queries — never from request body

## Follow-Up Notes

**Write-off % approximation:** Write-off percentage is computed from billing pack `writeoff_value / (writeoff_value + recoverable_value)`. A dedicated write-off field per engagement would be more precise — tracked follow-up.

**`linked_deadline_id`, `linked_period_id`, `linked_billing_pack_id`:** Fields exist in the schema but no UI currently sets them. These are reserved for Codebox 21+ when direct record linking from the actions page is added.

**Action summary card:** `GET /actions/summary` is implemented but not yet wired to a dashboard card. Can be added to the practice dashboard in a future codebox.

**Practice Notifications + Reminder Center Foundation**

After risks and actions exist, the practice needs a central view for:
- Overdue health actions (due_date < today, status = open/in_progress)
- Deadlines due in the next 7 / 14 / 30 days
- Review tasks waiting longer than N days
- Billing packs in draft/reviewed state for > 30 days
- Queued periods past period_start

Central `/practice/reminders.html` with prioritised actionable items across all clients.
