# Codebox 46 — Practice Knowledge Base + Technical Opinion Library

> Module: Practice Management — Knowledge Base
> Status: Complete (migration 104 not yet applied to Supabase)
> Migration: 104
> Routes: `/api/practice/knowledge/*`

---

## Purpose

A central, human-controlled source of truth for:
- SARS interpretations
- Internal policies
- Technical opinions
- Standard operating procedures
- Working paper notes
- Practice decisions
- Tax treatment notes
- Client-specific technical positions

**This is NOT AI-generated advice. NOT Sean AI. NOT automatic tax interpretation.**
All content is authored, reviewed, and approved by practice staff. Articles can be
linked to clients, tax records, disputes, completion packs, workflows, tasks, and
document requests so relevant context surfaces where staff are already working.

### `practice_knowledge_articles` — one row per article

| Field | Notes |
|---|---|
| `category` | 14-value CHECK: `income_tax`, `company_tax`, `provisional_tax`, `vat`, `paye`, `payroll`, `cipc`, `coida`, `accounting`, `audit`, `secretarial`, `internal_policy`, `workflow`, `other` |

| `status` | 4-value machine: `draft → under_review → approved → archived` |

Auto-updates `updated_at` via `tg_pka_updated_at` trigger. `id` uses `GENERATED BY DEFAULT AS IDENTITY` (per spec, not `SERIAL`).

### `practice_knowledge_links` — article ↔ record links

`linked_type` (13-value CHECK): `client`, `taxpayer_profile`, `individual_tax_return`, `company_tax_return`, `provisional_tax_plan`, `tax_submission`, `sars_statement_line`, `tax_dispute`, `tax_completion_pack`, `workflow_run`, `task`, `document_request`, `compliance_pack`.

`UNIQUE (article_id, linked_type, linked_id)` prevents duplicate links at the DB level (backed by an app-level 409 on duplicate).

### `practice_knowledge_events` — append-only audit log

Never updated or deleted.

## Backend — `knowledge-base.js`

### Endpoints (14 total)

| Method | Route | Purpose |
|---|---|---|
| GET | `/summary` | Status counts (draft/under_review/approved/archived) |
| GET | `/linked/:linkedType/:linkedId` | Articles linked to a given record — used by integration links |
| GET | `/` | Paginated list with filters: search, category, article_type, status, tag |
| POST | `/` | Create article (always starts as `draft`) |
| GET | `/:id` | Single article |
| PUT | `/:id` | Update — see versioning rule below |
| DELETE | `/:id` | Soft archive (alias of `/archive`) |
| PUT | `/:id/submit-review` | Status: draft → under_review |
| PUT | `/:id/approve` | Status: under_review → approved; sets reviewed/approved by+at |
| PUT | `/:id/archive` | Status: any non-terminal → archived |
| GET | `/:id/links` | List linked records |
| POST | `/:id/links` | Link to a record (ownership-checked where the table is known) |
| DELETE | `/:id/links/:linkId` | Remove a link |
| GET | `/:id/events` | Append-only event log |

Routes ordered so `/summary` and `/linked/...` are matched before `/:id` (Express path-collision avoidance, same convention as tax-disputes.js / tax-completion.js).

### State Machine

```
draft → under_review → approved → archived
  (any non-archived) → archived
```

Only `archived` is terminal. `PUT /:id` and `POST /:id/links` are blocked once archived.

### Versioning Rule (low-risk, no full version history table)

Editing an **approved** article's content-affecting fields (`title`, `summary`, `content`, `category`, `article_type`) via `PUT /:id`:
- Bumps `version` by 1
- Resets `status` to `draft`
- Clears `approved_at` / `approved_by`
- Requires the article to go through `submit-review` → `approve` again

Editing a `draft` or `under_review` article does not bump the version or change status.

### Linked-Record Ownership Check

`_verifyLinkedRecordOwnership` maps all 13 `linked_type` values to their known practice tables (`practice_clients`, `practice_taxpayer_profiles`, `practice_individual_tax_returns`, `practice_company_tax_returns`, `practice_provisional_tax_plans`, `practice_tax_submissions`, `practice_sars_statement_lines`, `practice_tax_dispute_cases`, `practice_tax_completion_packs`, `practice_workflow_runs`, `practice_tasks`, `practice_document_requests`, `practice_compliance_packs`) and confirms the record belongs to the requesting company before allowing a link. If a mapped table doesn't contain the given ID for this company, `POST /:id/links` returns 404.

### Search Logic

`ilike` on `title`, `summary`, `content`, `source_reference` (Supabase `.or()` with `ilike.%term%`). No AI search. Tag filter is a simple case-insensitive exact match against the JSONB `tags` array, done in-memory after the DB query (per spec: "simple contains fallback, not AI search").

### Multi-Tenant Safety

Every query scoped to `req.companyId`. `_verifyArticle` and `_verifyLink` re-check ownership before every mutating action. Linked-record ownership verified before creating a link (for known tables).

## Integration

### `index.js` Mount

```javascript
const knowledgeBaseRouter = require('./knowledge-base');
router.use('/knowledge', knowledgeBaseRouter);
```

Mounted after `/tax-completion`, before the `/dashboard` sub-router block.

### `layout.js` Nav

"Knowledge Base" entry added between "Tax Completion" and "Tax Config".

### Small Integration Links (low-risk, additive only)

A **"Knowledge ↗"** link (purple, `#2d1e4d` / `#b794f4`) was appended to the existing detail-footer of three pages, following the exact pattern used for the Codebox 44/45 "Open Disputes ↗" / "Completion Pack ↗" links — no existing buttons were changed:

- `tax-disputes.js` → links to `?linked_type=tax_dispute&linked_id=<case id>`
- `tax-completion.js` → links to `?linked_type=tax_completion_pack&linked_id=<pack id>`
- `tax-submissions.js` → links to `?linked_type=tax_submission&linked_id=<submission id>`

On `knowledge-base.html`, `?linked_type=X&linked_id=Y` renders an info banner above the table listing articles already linked to that record (via `GET /linked/:linkedType/:linkedId`), with each title clickable to open its detail modal.

## Frontend — `knowledge-base.html` + `js/knowledge-base.js`

### Page

- Summary cards (Draft, Under Review, Approved, Archived) — clickable to filter
- Filter bar: category, type, status, tag, free-text search
- Paginated article table (type, status, title, category, tags, version, updated)
- Create Article modal (title, category, type, summary, content, tags, source reference, effective from/to, internal notes) — always creates as `draft`
- Article Detail modal — 3 tabs: **Content** (full metadata + summary + content + internal notes), **Links** (linked records + add/remove), **Events** (append-only log)
- Edit modal — shown from detail footer; warns before saving if the article is `approved` (edit will create a new version and return it to draft)
- Link Article modal — select `linked_type` + record ID + notes
- Context-sensitive footer: draft → Submit for Review; under_review → Approve; both → Edit + Archive

### No localStorage / KV

Zero browser storage usage. All state from server. `node --check` passed on both new JS files.
