# Codebox 19 — Client Risk Scoring + Service Health Foundation

**App:** Lorenco Practice Management  
**Codebox:** 19 of ±80  
**Status:** Complete  
**Date:** 2026-06-20  

---

## Purpose

Provide a transparent, rule-based client health scoring system so that practice managers can see at a glance which clients are operationally risky. Based entirely on observable counts from the database. No AI, no predictive modelling, no automated decision-making.

---

### Backend (`backend/modules/practice/client-health.js`)

4 endpoints:

| Method | Path | Purpose |
|--------|------|---------|
| GET | `/api/practice/client-health/summary` | Status counts (total, good, watch, at_risk, critical, unknown) |
| GET | `/api/practice/client-health` | Paginated list with cached health fields + live aggregates |
| GET | `/api/practice/client-health/:clientId` | Fresh on-demand score for one client |
| POST | `/api/practice/client-health/recalculate` | Persist score(s) to DB + snapshot table |

### Scoring Rules

Start at 100. Apply deductions. Clamp at 0. If zero activity → `score = null`, `status = unknown`.

| Rule | Deduction |
|------|-----------|
| Any overdue deadline (due_date < today, status not closed) | −25 |
| More than 3 overdue deadlines | −15 (cumulative) |
| Pending review tasks > 5 | −15 |
| Unassigned active tasks > 0 | −10 |
| Queued periods past period_start | −15 |
| Active engagement with no responsible_team_member_id | −10 |
| Recurring engagement missing recurrence_start_date or recurrence_day | −10 |
| WIP billing pack older than 30 days | −15 |
| Write-off percentage > 20% | −10 |
| Client has no responsible_team_member_id | −10 |

### Status Thresholds

| Score | Status |
|-------|--------|
| 85–100 | good |
| 65–84 | watch |
| 40–64 | at_risk |
| 0–39 | critical |
| null | unknown (no activity data) |

## Multi-Tenant Safety

Every DB query uses `.eq('company_id', req.companyId)` sourced from the JWT. No company ID ever comes from the request body. The `fetchHealthData()` function passes `cid` from `req.companyId` to all 7 parallel queries.

---

## Rule D Compliance

No business data written to localStorage, sessionStorage, or KV. The health score is stored in the `practice_clients` table and `practice_client_health_snapshots` table. Frontend reads from the API only.

---

## Efficiency Pattern

For `POST /recalculate` without a specific `client_id`:
- `fetchHealthData(cid, null)` runs 7 parallel queries for the entire company
- `scoreClientFromData(client, data)` filters those arrays per client in JS
- Total: 7 Supabase queries regardless of N clients (not 7 × N)
