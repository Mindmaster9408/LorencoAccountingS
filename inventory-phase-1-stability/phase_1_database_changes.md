## Summary

One migration file applies four distinct database changes. All changes are safe to apply on a live database without downtime. All ALTER TABLE statements use `IF NOT EXISTS` or equivalent idempotent patterns.

---

## Change 1: Canonical Supplier Columns

**Table affected:** `suppliers`

**Problem:** The POS module created the `suppliers` table with column names that differed from the inventory backend's expectations.

| Legacy column (POS) | Canonical column (inventory backend) |
|---------------------|--------------------------------------|
| `supplier_name` | `name` |
| `contact_email` | `email` |
| `contact_phone` | `phone` |
| `tax_reference` | `vat_number` |
| *(absent)* | `notes` |

**Changes applied:**

```sql
ALTER TABLE suppliers
  ADD COLUMN IF NOT EXISTS name        VARCHAR(255),
  ADD COLUMN IF NOT EXISTS email       VARCHAR(255),
  ADD COLUMN IF NOT EXISTS phone       VARCHAR(50),
  ADD COLUMN IF NOT EXISTS vat_number  VARCHAR(50),
  ADD COLUMN IF NOT EXISTS notes       TEXT;

UPDATE suppliers SET
  name       = COALESCE(name, supplier_name),
  email      = COALESCE(email, contact_email),
  phone      = COALESCE(phone, contact_phone),
  vat_number = COALESCE(vat_number, tax_reference)
WHERE name IS NULL OR name = '';

UPDATE suppliers SET name = COALESCE(supplier_name, 'Unknown Supplier')
WHERE name IS NULL OR name = '';

ALTER TABLE suppliers ALTER COLUMN name SET NOT NULL;
```

**Safety:**
- `ADD COLUMN IF NOT EXISTS` is idempotent — safe to re-run
- Old columns (`supplier_name`, `contact_email`, `contact_phone`, `tax_reference`) are NOT dropped
- `COALESCE` pattern means the backfill only applies when the new column is null
- A fallback of `'Unknown Supplier'` prevents the `NOT NULL` constraint from failing on rows with no legacy name

## Change 2: Negative Stock Protection Constraint

**Table affected:** `inventory_items`

**Problem:** No database constraint prevented `current_stock` from going negative, which could happen through the non-atomic read-then-write pattern or direct DB manipulation.

**Changes applied:**

```sql
ALTER TABLE inventory_items
  ADD CONSTRAINT chk_current_stock_non_negative
  CHECK (current_stock >= 0) NOT VALID;
```

**Safety:**
- `NOT VALID` adds the constraint without scanning existing rows — non-blocking on a live table
- New writes and updates are constrained immediately
- The constraint is enforced at the Postgres level, providing a hard backstop in addition to the API-level validation in the RPC function
- A commented-out `VALIDATE CONSTRAINT` statement is in the migration for use after cleaning up any existing negative-stock rows

**Production checklist before validating:**
```sql
-- Run this first:
SELECT id, name, current_stock FROM inventory_items WHERE current_stock < 0;
-- Fix any rows returned before uncommenting VALIDATE CONSTRAINT
```

## Change 3: Purchase Orders Status Constraint Update

**Table affected:** `purchase_orders`

**Problem:** The original status constraint (from migration 007) only allowed `'partial'` as the partial-receipt status. The inventory backend and frontend were designed to use `'partial_receipt'` (matching the ecosystem's underscore-separated naming convention).

**Changes applied:**

```sql
-- Auto-detect and drop old status constraint (name is auto-generated by Postgres)
DO $$ ... DROP CONSTRAINT ... $$;

ALTER TABLE purchase_orders
  ADD CONSTRAINT purchase_orders_status_check
  CHECK (status IN ('draft', 'sent', 'partial_receipt', 'received', 'cancelled'));
```

**Safety:**
- The DO block detects the constraint by inspecting `pg_constraint` for any check on `purchase_orders` containing `status` — this handles both named and auto-named constraints
- The new constraint is named explicitly (`purchase_orders_status_check`) for easy reference

**Manual data fix (if old 'partial' rows exist):**
```sql
UPDATE purchase_orders SET status = 'partial_receipt' WHERE status = 'partial';
```
This should be run manually before applying the migration if any `partial` rows exist.

## Change 4: `adjust_inventory_stock()` Function

**Type:** New Postgres function  
**Language:** PL/pgSQL

**Purpose:** Single source of truth for all stock mutations. Replaces three separate non-atomic read-then-write patterns in the application code.

**Signature:**
```sql
CREATE OR REPLACE FUNCTION adjust_inventory_stock(
  p_company_id    INTEGER,
  p_item_id       INTEGER,
  p_delta         NUMERIC,       -- positive = stock in, negative = stock out
  p_movement_type VARCHAR(50),   -- 'in'|'out'|'return'|'adjustment'|'transfer'
  p_warehouse_id  INTEGER  DEFAULT NULL,
  p_reference     VARCHAR(255) DEFAULT NULL,
  p_notes         TEXT     DEFAULT NULL,
  p_cost_price    NUMERIC  DEFAULT NULL,
  p_created_by    INTEGER  DEFAULT NULL
)
RETURNS JSONB
```

**Return values:**

| Scenario | Return |
|----------|--------|
| Success | `{ "success": true, "new_stock": <number> }` |
| Insufficient stock | `{ "success": false, "error": "Insufficient stock", "available": <number> }` |
| Item not found | `{ "success": false, "error": "Item not found" }` |

**What it does atomically (within one transaction):**

1. `UPDATE inventory_items SET current_stock = current_stock + p_delta WHERE id = p_item_id AND company_id = p_company_id AND (current_stock + p_delta) >= 0`
2. If the UPDATE returns 0 rows → determine whether item doesn't exist or stock is insufficient → return error JSON
3. If the UPDATE succeeds → `INSERT INTO stock_movements (...)` with `ABS(p_delta)` as the quantity
4. Return success JSON with `new_stock`

**Multi-tenant safety:** `p_company_id` is a required parameter. The `WHERE company_id = p_company_id` clause is present on both the UPDATE and the EXISTS check.

**Safety:**
- `CREATE OR REPLACE` — safe to re-run
- The `NOT FOUND` branch means the `INSERT INTO stock_movements` only runs if the `UPDATE` succeeded — movement and stock are always in sync
- `ABS(p_delta)` ensures movement records always store positive quantities in the ledger, regardless of whether it was a deduction or addition

## Rollback Notes

There is no automated rollback script. If changes need to be reverted:

1. **Supplier columns**: `ALTER TABLE suppliers DROP COLUMN IF EXISTS name, email, phone, vat_number, notes;`
2. **Stock constraint**: `ALTER TABLE inventory_items DROP CONSTRAINT IF EXISTS chk_current_stock_non_negative;`
3. **PO status constraint**: Drop `purchase_orders_status_check` and re-add the original constraint with `'partial'` instead of `'partial_receipt'`
4. **RPC function**: `DROP FUNCTION IF EXISTS adjust_inventory_stock;`

Rolling back the supplier columns and re-deploying the old backend code would restore the original (broken) behaviour. **Not recommended** — the original behaviour was the bug.
